# Navigation and SLAM with Isaac ROS

Once your robot is in simulation, the next critical task is to give it autonomy. This means it needs to know where it is (**Localization**), what the world looks like (**Mapping**), and how to get from A to B (**Path Planning**).

## SLAM: Simultaneous Localization and Mapping

**SLAM** is the core problem of building a map of an unknown environment while simultaneously keeping track of the robot's position within that map.

-   **Visual SLAM (VSLAM):** A specific type of SLAM that uses camera data (visual features) to build the map and localize. This is computationally intensive but provides rich information about the environment.
-   **Isaac ROS VSLAM:** NVIDIA provides hardware-accelerated ROS 2 packages for VSLAM, designed to run efficiently on NVIDIA GPUs. These packages can process camera data in real-time to generate a map and estimate the robot's pose (position and orientation).

## The Navigation 2 Stack (Nav2)

**Nav2** is the standard navigation stack in ROS 2. It's a highly configurable system that takes sensor data and a goal pose, and outputs velocity commands to move the robot.

It is composed of several key servers:
1.  **Map Server:** Loads a previously created map or receives a live map from a SLAM system.
2.  **AMCL (Adaptive Monte Carlo Localization):** A localization system that uses a particle filter to estimate the robot's position on a known map.
3.  **BT Navigator (Behavior Tree Navigator):** The high-level planner. It uses a Behavior Tree to orchestrate the entire navigation process, from computing a path to a goal to performing recovery behaviors if the robot gets stuck.
4.  **Global Planner:** Computes the overall path from the robot's current location to the goal pose, avoiding known obstacles from the static map.
5.  **Local Planner / Controller:** Computes safe velocity commands to follow the global plan while avoiding immediate, dynamic obstacles detected by sensors.

### Navigation for Humanoids

While Nav2 was originally designed for wheeled robots, its principles can be adapted for legged robots, including humanoids. The key difference is that the output of the Nav2 stack (`/cmd_vel` messages) cannot be sent directly to wheel motors. Instead, it must be interpreted by a **walking controller** or a **gait planner**.

This controller translates the desired linear and angular velocities into the complex sequence of joint torques and foot placements required for stable bipedal locomotion. This is where reinforcement learning policies trained in Isaac Gym often come into play.
