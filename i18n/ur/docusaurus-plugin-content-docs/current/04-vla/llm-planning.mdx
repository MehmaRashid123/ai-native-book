# LLMs کو بطور High-Level Planners استعمال کرنا

ایک transcribed voice command جیسے "robot, clean the room" اب بھی ایک روبوٹ کے لیے براہ راست عمل کرنے کے لیے بہت abstract ہے۔ روبوٹ کا کنٹرول سسٹم concrete actions کو سمجھتا ہے جیسے "move joint A to 45 degrees" یا "navigate to coordinates (x, y)"۔

یہ وہ جگہ ہے جہاں ایک Large Language Model (LLM) جیسے GPT-4o ایک **high-level cognitive planner** کے طور پر کام کر سکتا ہے۔ اس کا کام abstract انسانی کمانڈ کو concrete، executable sub-tasks کی ایک sequence میں decompose کرنا ہے جسے روبوٹ کے lower-level systems سنبھال سکیں۔

## The LLM Planner Pattern

یہ عمل عام طور پر اس طرح نظر آتا ہے:

1.  **System Prompt Engineering:** آپ LLM کو ایک detailed system prompt فراہم کرتے ہیں جو اسے اس کے کردار، روبوٹ کی صلاحیتوں اور دستیاب tools کے بارے میں بتاتا ہے۔

    > **System Prompt Example:**
    > "You are a helpful robot assistant. Your task is to break down user commands into a sequence of functions. You have access to the following functions: `navigate_to(location)`, `find_object(object_name)`, `pick_up(object_id)`, and `place_at(location)`. The only known locations are 'kitchen', 'living_room', and 'recycling_bin'."

2.  **User Command:** آپ transcribed user command کو LLM کے پاس بھیجتے ہیں۔

    > **User Command:** "please pick up the empty soda can and throw it away"

3.  **LLM Function Calling:** LLM، اپنی reasoning capabilities اور system prompt سے context کا استعمال کرتے ہوئے، ایک structured plan تیار کرتا ہے۔ Modern LLMs اسے JSON object کے طور پر output کر سکتے ہیں جو function calls کی sequence کی نمائندگی کرتا ہے۔

    > **LLM Output (as a plan):**
    > ```json
    > [
    >   { "function": "navigate_to", "params": { "location": "living_room" } },
    >   { "function": "find_object", "params": { "object_name": "soda_can" } },
    >   { "function": "pick_up", "params": { "object_id": "soda_can_id_123" } },
    >   { "function": "navigate_to", "params": { "location": "recycling_bin" } },
    >   { "function": "place_at", "params": { "location": "recycling_bin" } }
    > ]
    > ```
    > *(Note: `object_id` کو `find_object` step کے output سے fill کیا جائے گا۔)*

4.  **Plan Execution:** آپ کے کوڈ میں ایک separate "plan execution" module اس JSON plan کو وصول کرتا ہے۔ پھر یہ steps کے ذریعے iterate کرتا ہے، اور ہر ایک کے لیے متعلقہ ROS 2 services یا actions کو call کرتا ہے۔
    -   یہ `navigate_to` ROS 2 action server کو call کرتا ہے۔
    -   ایک بار جب وہ کامیاب ہو جاتا ہے، تو یہ `find_object` service کو call کرتا ہے۔
    -   اور اسی طرح، جب تک کہ plan مکمل نہ ہو جائے یا کوئی step fail نہ ہو جائے۔

یہ architecture concerns کی ایک powerful separation پیدا کرتا ہے۔ LLM "what" (the plan) کو سنبھالتا ہے، اور روبوٹ کا ROS 2-based کنٹرول سسٹم "how" (the physical execution) کو سنبھالتا ہے۔
